Ryzyko kredytowe

Kada instytucja udzielajca kredytu musi szacowa zdolno kredytow i prawdopodobiestwo, czy dana osoba spaci kredyt. W praktyce wikszo osb paci prawidowo, wic zamiast tego przewiduje si prawdopodobiestwo, e kto bdzie mia opnienia w patnoci w okrelonym czasie, tzw. probability of default. Jak sama nazwa wskazuje, nie jest to klasyfikacja binarna, tylko faktyczne prawdopodobiestwo, wykorzystywane przy ocenie ryzyka.

Wykorzystamy zbir z konkursu Give Me Some Credit. Jego zmienne to (rdo):

  • SeriousDlqin2yrs - brak spaty raty kredytu przez 90 dni lub duej w cigu ostatnich 2 lat; s to nasze klasy y
  • RevolvingUtilizationOfUnsecuredLines - aktualne zapoyczenie na kartach kredytowych i poyczkach w stosunku do maksymalnego moliwego zapoyczenia
  • age - wiek w latach
  • DebtRatio - aktualne cakowite zapoyczenie wzgldem miesicznego dochodu
  • MonthlyIncome - miesiczne dochody
  • NumberOfOpenCreditLinesAndLoans - liczba aktualnych poyczek i kart kredytowych
  • NumberOfTimes90DaysLate - ile razy w przeszoci wystpio opnienie 90 dni lub duej w spacie raty kredytu
  • NumberRealEstateLoansOrLines - liczba poyczek z nieruchomoci jako zabezpieczeniem
  • NumberOfDependents - liczba pozostaych osb w gospodarstwie domowym
  • NumberOfTime30-59DaysPastDueNotWorse - ile razy w cigu ostatnich 2 lat nastpio opnienie w spacie raty midzy 30 a 59 dni
  • NumberOfTime60-89DaysPastDueNotWorse - jak wyej, ale 60-89 dni

Ze wzgldu na potrzeb precyzyjnych prawdopodobiestw oraz rozmiar danych (banki maj ogromn liczb klientw), regresja logistyczna to idealny wybr dla tego zadania. Metryk w takich przypadkach jest AUROC (Area Under Receiver Operating Characteristic curve), bo bierze ona pod uwag prawdopodobiestwo klasy pozytywnej.

Na pocztek wczytamy i przeanalizujemy wstpnie nasze dane. Jako zbir testowy wykorzystamy 20% danych. Jako e mamy do czynienia z klasyfikacj niezbalansowan, to wykorzystamy podzia ze stratyfikacj (stratified split), w ktrym proporcje klasy pozytywnej i negatywnej zostaj zachowane w zbiorze treningowym i testowym.

Faktycznie w porwnanie z lasem losowym, spodziewabym si wikszej rznicy na korzy regresji

Sprawdmy jeszcze, jak dobrze nam idzie z predykcjami prawdopodobiestwa. Regresja logistyczna jest z natury dobrze skalibrowana, ale nie ma takiej gwarancji na kadym zbiorze danych. W szczeglnoci moe mie problemy na zbiorach niezbalansowanych, przez bardzo niewielk ilo klasy pozytywnej (a priori jest rzadka).

Do rysowania krzywych kalibracji suy klasa CalibrationDisplay. W perfekcyjnym przypadku prawdopodobiestwa modelu rosn liniowo raz z "pewnoci", e punkt naley do klasy pozytywnej. Dla zainteresowanych, dokumentacja Scikit-learn omawia te krzywe nieco dokadniej.

Detekcja spamu

(rdo: "Spam" sketch, Monty Python's Flying Circus)

Wykorzystamy teraz klasyfikator Naiwnego Bayesa do klasyfikacji binarnej tekstw - czy email jest spamem, czy te nie (tzw. "ham"). Takie zadanie nazywa si te spam/ham classification. Zbiorem bdzie Enron Spam dataset, zebrany na potrzeby artykuu:

Metsis, Vangelis, Ion Androutsopoulos, and Georgios Paliouras. "Spam filtering with Naive Bayes - which Naive Bayes?." CEAS. Vol. 17. 2006. link

Upadek firmy energetycznej Enron i skandal wywoany wiadomociami o nieprawidowociach finansowych spowodoway, e podczas ledztwa i procesu zmuszono firm do upublicznienia penej zawartoci skrzynek mailowych 150 pracownikw. Jest to doskonae rdo prawdziwych danych, typowych emaili niebdcych spamem, i byo wykorzystywane wielokrotnie na potrzeby tworzenia wielu rnych zbiorw danych.   5. Dla kadego z przykadw, dla kadej z klas, sumujemy log-prior i log-likelihood, otrzymujc warto rozkadu a posteriori (posterior). Zgodnie z zasad maximum a posteriori (MAP), wybieramy klas o najwyszej z tych wartoci. Moe tu si przyda np.argmax().

from sklearn.base import BaseEstimator, ClassifierMixin
from sklearn.neighbors import KernelDensity
from sklearn.utils import check_array, check_X_y


class KernelDensityEstimationNB(BaseEstimator, ClassifierMixin):
    """
    Naive Bayes with KDE estimation of feature distributions.

    Parameters
    ----------
    kernel : str
        Kernel name, passed to KernelDensity

    bandwidth : float
        Kernel bandwidth
    """

    def __init__(self, kernel: str = "gaussian", bandwidth: float = 1.0):
        # create attributes
        self.kernel = kernel
        self.bandwidth = bandwidth

    def fit(self, X, y):
        # make sure that we are working on Numpy arrays, and perform basic
        # sanity checks
        X = np.array(X)
        y = np.array(y)
        X, y = check_X_y(X, y)

        # make sure that y has values from 0 to num_classes - 1
        assert y.min() == 0
        assert y.max() == len(np.unique(y)) - 1

        # vector of possible classes
        self.classes_ = np.sort(np.unique(y))

        # vector for remembering class priors, i.e. proportion of classes
        # in the training set
        self.log_priors_ = np.empty(len(self.classes_))

        # list of KernelDensity models, one for each class
        self.kde_models_ = []

        for class_idx in self.classes_:
            # select X and y from the current class
            class_mask = (y == class_idx)
            X_class = X[class_mask]
            y_class = y[class_mask]

            # calculate log-prior, i.e. logarithm of current class proportion
            self.log_priors_[class_idx] = np.log(X_class.shape[0] / X.shape[0])

            # create and fit KernelDensity estimator, using given kernel and bandwidth
            kde = KernelDensity(kernel=self.kernel, bandwidth=self.bandwidth)

            kde.fit(X_class)

            # save KDE
            self.kde_models_.append(kde)

        return self

    def predict(self, X):
        # make sure that we are working on Numpy array, and perform basic
        # sanity checks
        X = np.array(X)
        X = check_array(X)

        # use KDE models and calculate log-likelihoods for each sample
        log_likelihoods = np.array([kde.score_samples(X) for kde in self.kde_models_]).T

        # calculate posterior probability: log-prior + log-likelihood
        # make sure that they have the same shape (transpose if needed)
        posteriors = self.log_priors_ + log_likelihoods

        # select the index of class with maximum a posteriori (MAP) rule
        map_class = self.classes_[np.argmax(posteriors, axis=1)]

        return map_class

W przypadku czuoci (recall) jest ju zdecydowanie lepiej! F1-score poprawi si, ale tylko troch - precyzja mogaby by lepsza. Ale KDE jest do czue na dobr szerokoci (bandwidth) kernela, wic tuning hiperparametrw powinien tutaj pomc.

Ciekaw opcj dla szerokoci kernela jest Silverman's rule of thumb. Jest to prosty wzr na szeroko kernela przy zaoeniu, e rozkad jest unimodalny. Scikit-learn wspiera t opcj przez przekazanie wartoci bandwidth="silverman". Wzr dla wielu cech (Wikipedia):

$$ h_{silverman} = \left(\frac{4}{d+2}\right) n^{\frac{-1}{d+4}} \sigma $$

Gdzie $\sigma$ to wektor odchyle standardowych cech, $n$ to liczba prbek, a $d$ to liczba wymiarw.

Wad implementacji w Scikit-learn jest to, e nie uwzgldnia on odchyle $\sigma$, wic bandwidth jest wsplny dla wszystkich cech. Jeeli jednak s one w podobnej skali (tak jak w tym zbiorze), to nie stanowi to duego problemu. No i w razie potrzeby mona zawsze ustandardyzowa dane.

Dla zainteresowanych - tuning kernel bandwidth to mocno badane przez matematykw zagadnienie, i istnieje do niego szereg rozbudowanych metod statystycznych. Ten tutorial jest chyba najlepszym i najbardziej czytelnym, a przy tym ma potny przegld literatury.

Najpopularniejsz technik nieparametryczn, ktra pozwala oszacowa szeroko kernela, jest estymator Sheathera-Jonesa (Sheather-Jones estimator). Uywa on do zaawansowanej statystyki, aby zminimalizowa bd redniokwadratowy midzy oszacowaniem KDE a prawdziwym (nieznanym) rozkadem. Typowo daje duo lepsze wyniki od reguy Silvermana, bo wiele cech nie jest unimodalnych i/lub zblionych do rozkadu normalnego.

Dodatkowo KDE w postaci ze Scikit-learn, uywajce wprost definicji i opierajce si na najbliszych ssiadach, nie jest zbyt szybkie. Mona pokaza, e sumowanie kerneli z kadego punktu odpowiada operacji konwolucji (convolution), a dziki temu mona wykorzysta twierdzenie o splocie (convolution theorem) i obliczy to za pomoc szybkiej transformaty Fouriera (FFT). Algorytm FFTKDE jest rzdy wielkoci szybszy od tradycyjnego podejcia, i wietnie si skaluje.
